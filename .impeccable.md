# Impeccable — Prism

Design context for the Prism developer browser. This guides all UI/UX work.
Keep it current; update the Design Context section in place rather than appending.

## Design Context

### Users
Developers (initially the creator and people like him) doing **side-by-side A/B
and responsive testing**. They run several real Chromium viewports at once,
mirror one set of interactions across them, compare routes, and diff network
traffic between panes. Sessions are long and often late at night, with bright
web content rendered inside the panes. They already know what they're doing — the
tool should reward expertise, not hand-hold. The interface is *chrome around
other people's pages*, so it must never fight the content it frames.

### Brand Personality
**Crafted, warm, opinionated.** A precision instrument that someone clearly made
by hand and cares about — closer to a well-machined measuring tool than a SaaS
dashboard. Quiet and exact in normal use, with a point of view in the details.
Emotional goal: *calm confidence*. The user should trust every reading on screen
(dimensions, diffs, proxy state) the way you trust a good caliper.

### Aesthetic Direction
**Precision instrument**, executed warm rather than clinical.

- Tight modular grid, hairline (1px) rules instead of heavy borders or cards,
  generous internal calm despite high information density.
- **Tabular / monospaced numerics** for everything measured: viewport
  dimensions, status codes, sizes, timings. Numbers should align in columns and
  read like instrument readouts. (Mono is for *data*, never as generic
  "developer" decoration on labels or prose.)
- Theme: **dark is primary** (chrome recedes so rendered pages pop; comfortable
  for late sessions). Support **light** via `light-dark()` as a real second
  theme, not an afterthought.
- **Single warm signal accent.** Neutrals are near-monochrome and *tinted toward
  the brand hue* (a warm brass/amber — think aged instrument metal, not retro
  phosphor). Color is spent almost entirely on *meaning*: live/active state,
  network diffs, proxy on/off. A rare accent hits hard because it is rare.
- Semantic colors are part of "color = meaning": a restrained warm-leaning red
  for diffs/mismatch, a calm green for proxy-up. Keep them muted; they are
  signal, not decoration.

References (feel, not copying): fine measuring instruments, aviation/marine
gauges, Teenage Engineering's restraint, Braun product graphics.
**Anti-references:** the generic VSCode-clone dark dev UI; the AI palette
(cyan-on-dark, purple→blue gradients, neon glows); glassmorphism; rounded cards
with soft drop shadows; consumer playfulness.

### Hard Constraints
- **Stack:** Electron + **Svelte 5** (runes) renderer; styles in
  `src/renderer/src/styles.css` + scoped component styles. No CSS framework.
- **No emoji** anywhere in UI.
- **Icons: Phosphor** (`phosphor-svelte`), **light** weight as the default to
  match the hairline/instrument feel (step to regular only where an icon must
  carry weight, e.g. an active toggle). Rules: import through one shared icon
  module (e.g. `src/renderer/src/lib/icons.ts`) and reuse — never inline `<svg>`
  at call sites, never use unicode/emoji glyphs as icons. Icons inherit
  `currentColor` and size in `em` so they track the text/control they sit in.
  These replace the current unicode glyphs (`‹ › ↻ ✕ ⌥⌘I`, the ⚙ gear, etc.).
- Never use em dashes in UI copy.
- The chrome frames live web pages — it must stay visually subordinate to pane
  content and never tint or overlay it.

### Design Principles
1. **Instrument, not interface.** Every element should read as a precise
   control or readout. If it doesn't measure, control, or label something, cut it.
2. **Color is signal.** Warm-tinted near-monochrome by default; spend hue only on
   state and difference. If everything is colored, nothing is.
3. **Numbers are first-class.** Anything measured is tabular and column-aligned,
   legible at a glance like a gauge.
4. **Hairlines over boxes.** Structure with 1px rules, alignment, and spacing
   rhythm — not cards, heavy borders, or shadows. (No side-stripe accents ever.)
5. **Subordinate chrome.** The UI frames other people's pages; it stays calm and
   recedes so the content is the loudest thing on screen.

## Typography & token notes (decided at craft time)
- Run the font-selection procedure per the skill each build; **avoid** the reflex
  list. Pair a warm, slightly opinionated label/display face with a precise body
  face, plus a true tabular mono for numerics. Mono is for data only.
- Tokens: OKLCH throughout, 4pt spacing scale with semantic names, neutrals
  tinted toward the brand hue, reduced chroma near white/black.
